## Architecture

```
Author Input (phonics skill, level, settings)
       ↓
OpenAI API → Structured Curriculum Objects (JSON)
       ↓
Supabase DB (persist objects)
       ↓
Editor Interface (author edits objects directly)
       ↓
PDF Template Engine → Publication-ready PDF
       ↓
Supabase Storage (store exported PDFs)
```

**Core principle:** AI generates structured data (curriculum objects), never final pages. Templates render all output. This keeps every field editable and exports visually consistent.

## Conventions

- App Router structure: `app/` for routes, `components/` for UI, `lib/` for utilities and API clients, `types/` for TypeScript interfaces
- Supabase client: server-side via `@supabase/ssr`, client-side via `createBrowserClient`
- All AI prompts stored as typed prompt-builder functions in `lib/prompts/`
- Curriculum data types defined in `types/curriculum.ts` — mirrors DB schema exactly
- Component naming: PascalCase; file naming: kebab-case

## Environment & Setup

Required environment variables:
```
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
SUPABASE_SERVICE_ROLE_KEY=
OPENAI_API_KEY=
```

Local dev:
```bash
npm install
npm run dev
```

Deploy: push to `main` → Vercel auto-deploys. Set env vars in Vercel dashboard.

## Known Constraints / Gotchas

- AI output must be validated against expected schema before persisting — OpenAI JSON mode helps but doesn't guarantee structure
- Every vocabulary word requires an image before publishing — enforce this at the export step, not at generation
- PDF rendering library choice has significant architectural implications: `@react-pdf/renderer` runs in Node, no browser needed; Puppeteer requires a headless browser and more memory on Vercel
- Supabase free tier has storage limits — monitor if vocabulary image volume grows
- Language transfer notes are Spanish-focused but must not make the curriculum Spanish-exclusive; prompts must enforce this distinction
